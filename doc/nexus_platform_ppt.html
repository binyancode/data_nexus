Data Nexus

在业务语义与物理数据之间，架一座桥

一、要解决什么问题

现状：一个业务问题的答案散落在数据库、文档、报表等多个系统，需要人工逐个查再拼；目标：一句话提问，系统自动跨源取数、归因，并在需要时直接执行动作

二、分层架构：Data Nexus 逻辑上分几层

设计思路：五层各司其职、只依赖下一层接口；权限与血缘作为竖切面贯穿全层，而非独立成层。后面各页就是逐层展开

三、系统蓝图：所有部件如何联合运作

上层知识、下层能力是两块静态底座，中间运行引擎是动态流水线；一次提问自上而下贯穿三层，后续每页展开其中一块

四、Concept：业务名词长什么样

五种 kind（entity/attribute/relation/metric/derivation）共用同一外壳；物理只落 entity 和 attribute，其余纯语义——概念只用 concept_id 互相引用，不出现物理表

五、Binding：概念怎么对接物理表

Binding 只把 entity→表、attribute→列；join 不在 Binding 里，由 relation 声明；换库或加源只改 Binding，Concept 不变

六、SQG（Semantic Query Graph，语义查询图）：把问题翻成的统一查询指令

SQG 是编译器产出的逻辑 DAG（算子作用在「概念」上，不含 resolver、不含物理表）；一共 6 个算子，下面逐个说清楚各自干什么

七、Resolver：谁去执行（源 = 智能体 = 动作）

数据源、智能体、动作统一成同一个执行接口；优化器眼中不分类型，只有一批把查询变成结果的能力单元

八、Resolver 源类清单

关系库、向量库、AI 智能体（Microsoft IQ：Fabric / Foundry / Web / Work IQ）、REST API、写回动作，甚至「人」（human-in-the-loop）都实现同一 Resolver 接口，被统一调度

九、Concept 和 Binding 怎么自动生成

每个源接入时都带一个 Resolver（适配器）；建本体就调它的 describe()/sample() 探出源结构与样例，机器据此自动推断候选，人工只需确认纠正

十、执行总览：一次提问，四段引擎跑完

例：「华东上季度毛利为何下滑、跟华南比差在哪？给份说明，并把复盘任务派给区域负责人。」——初始化器先备料（选本体 + 检索口径/示例，见下页），再交编译器 → 优化器 → 协调器 → 生成器，四段引擎流水线；跑完还有一层 Harness 评估回路，不合理就带着问题重来

十一、初始化器：编译前的智能备料

流水线第一段：在编译器动手前，先把「编译需要的一切」备好——选定本体、检索口径与示例、裁剪相关子图；把“备料”从编译器里拆出来，编译器只专注 NL→SQG

十二、编译器：把问题拆成语义查询图（逻辑 SQG）

编译器按 Concept 把自然语言翻成一张 DAG——6 个节点 + 依赖边；此刻只有「问什么、谁依赖谁」，还没决定「谁来答、怎么跑」，绝不含 Resolver 或 SQL

十三、优化器：逻辑 SQG → 物理执行计划（选源 · 下推 · 融合）

优化器是全流程的大脑：给每个节点选中一个 Resolver，把同源节点融成一条 SQL 一次下推、跨源节点各自调用、依赖别人的留占位符，并把当前用户透传到每个源

十四、协调器：按依赖分波并行 + 回填 + 合并裁决

优化器给了物理计划，协调器负责真正跑完这张 DAG——按依赖分波，能并行就并行；上游结果运行时回填下游；最后合并、数字冲突按信任分裁决

十五、生成器：把结果写成答案 + 血缘 + 回执

协调器合并后的结果交给生成器——组织成「对比结论 + 归因 + 证据」，标注每个数字的出处与信任分，并附上已完成的行动回执；全程留痕可回溯

十六、评估回路（Harness）：不合理就带着问题重来

四段引擎跑完不算完——外面还套一层 Harness：给整轮打个分（报错了吗？各节点输出合理吗？最终答案/动作靠谱吗？），不合理就把问题点连同原问题拼成新提示词、重新编译执行；外层循环有最大轮数上限，防止空转

十七、落地路径：分五步建，从简单到完整

先做最小的能用版本，再一步步加东西；每做完一步，都能单独演示一个用户用得上的功能

十八、价值：这套设计带来什么

因为所有东西都统一成同一套做法，顺带得到这五个好处
